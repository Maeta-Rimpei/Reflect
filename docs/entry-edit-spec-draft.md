# ふりかえり編集・日次分析のやり直し — 仕様まとめ

本書は **現時点で合意した仕様**と、実装時の参照メモ（影響範囲・未決）をまとめたもの。実装タスクは別とする。

---

## 1. 仕様（確定）

### 1.1 全体

| # | 内容 |
|---|------|
| 1 | **編集**と**日次 AI のやり直し**は **別操作**（編集は別 API）。編集では **Gemini を呼ばない**。 |
| 2 | **編集**は **全期間**のエントリを対象（`posted_at` は変えない）。**Free / Deep 共通**で編集可能。**編集保存時は既存の日次分析を削除する**（§1.6）。 |
| 3 | 日次の「やり直し」は **種類 A / B** に分ける（§1.2）。 |
| 4 | **種類 B（成功後の再分析）**は **Deep のみ**、**今日（東京）と同一の暦週**に属する日のエントリに限定し、**回数制限**を設ける（数値は別途）。 |
| 5 | **UI**: Free には **種類 B 不可**であることと **Deep のさりげない案内**。Deep には **回数制限の明示**。 |

### 1.2 種類 A / B（再分析の位置づけ）

| 種類 | 意味 | プラン | 目的のイメージ |
|------|------|--------|----------------|
| **A. 分析失敗時の再試行** | 日次分析が **一度も成功として永続化されていない**（未取得・失敗相当） | **Free / Deep 共通** | 初回生成の失敗・タイムアウト等の救済。従来の「AI分析を再試行」に近い |
| **B. 成功後の再分析** | 日次分析が **すでに成功として DB に存在する**状態で、あえて取り直す | **Deep のみ** | 本文修正後の付け直し等。**コスト抑制のため Deep + 同一週 + クォータ** |

### 1.3 判定ロジック（サーバー）

**意味論**: 種類 A は「**失敗したか／未取得か**」、種類 B は「**すでに成功した分析があるか**」。

**現行の実装（`POST /api/v1/entries`）の前提**（参考）:

- `entries` 保存後に Gemini を呼び、**成功時のみ** `analysis_results`（`type: daily`）へ insert する。
- トランザクションは **1本にまとまない**（別リクエスト）。**トランザクションは別々**。
- そのため **「日次分析が永続化されていない」＝失敗経路と実質一致**しやすい。

**判定の持ち方（実装メモ）**:

- **「永続化された成功があるか」** を見る。  
  - 多くの場合 **「`analysis_results` に当該エントリ／日の日次が有効に存在するか」** で代用できる（成功時だけ insert する前提）。
- **「空か」単体の話ではなく**、「**一度も成功として保存されていない**」＝ユーザー視点では **失敗／未取得** として種類 **A** としてよい。

**分岐**:

- **永続化された日次成功がない** → 種類 **A**。**プラン不問**。対象エントリは **§1.4 の同一暦週**に含まれる日に限る（**B と同じ週範囲**）。クォータは緩め or 別ルール（レート制限程度も可）。
- **ある** → 種類 **B**。**Deep かつ** `posted_at` の日付（東京）が **今日と同一の暦週に含まれる** **かつ** クォータ内のみ可（§1.4）。

### 1.4 種類 A / B の対象期間（同一暦週・東京）

- **種類 A・B 共通**: エントリの `posted_at` を東京の日付 `D` に正規化したとき、**`D` が「今日」と同一暦週に含まれる**ものだけが、再試行・再分析の **対象日**となる（**今日のみ**ではない）。
- **定義**: 週は **月曜始まり・日曜終わり**（**ISO 8601**）。`getWeekRangeInTokyo()` と同じ基準でよい。
- **種類 B 固有**: 上記に加え **Deep**・**クォータ**（§1.5）。
- **例**（今日が **金曜 3/20** の場合）:
  - 先週の **金 3/13** のふりかえり → **別週**のため、種類 **A・B とも対象日ではない**（その週の再試行・再分析は不可）。
  - 今週の **月 3/16** 〜 **金 3/20** → **同一週**のため対象日として **可**（A は未取得時・B は Deep＋クォータ等を満たす場合）。
- **直近 7 日ローリングとの違い**: ローリング 7 日だと「先週の後半」がまだ含まれることがあるが、**暦週**にすると **週の境界で一気に切れる**。説明が「**今週のふりかえりだけ**」としやすい。

### 1.5 種類 B の回数制限（汎用クォータテーブル）

**方針**: 種類 B 専用の1テーブルではなく、**将来ほかの機能でも「ユーザー×期間×回数制限」**を足しやすいよう、**小さな汎用クォータテーブル**を1本用意する（過剰なルールエンジンにはしない）。

- **カウント単位**: **ユーザー月次**（期間の基準は **東京の暦月**。`period` は例: `2026-03` 形式で保持）。
- **翌月の扱い**: **計上対象の月が切り替わる**（前月に上限まで使っても、翌月は **その月の枠が新しく**使える。利用回数は **0 から再カウント**）。「1 で上書き」ではなく **別月のレコード／別期間キー**として扱うイメージ。
- **DB テーブル**: `public.quota_usage`（`docs/supabase-migration.sql` に定義）
  - `user_id` — `users.id` への FK
  - `quota_key` — 制限の種類（日次再分析の月次上限は `monthly_daily_journal_regeneration` のような **固定文字列**。将来は週次レポート等で別キーを追加）
  - `period` — 集計期間（当面は東京の暦月 **`YYYY-MM`**）
  - `used_count` — その期間内の利用回数（種類 B 成功のたびに increment）
  - **PRIMARY KEY** `(user_id, quota_key, period)` — **1 ユーザー・1 キー・1 月に 1 行**
- **具体の N**（月あたり何回までか）はプロダクト決定。実装では **設定定数**または **将来の設定テーブル**と突き合わせてもよい。
- API は **429** + メッセージ + 可能なら `nextAvailable` を返す形を想定。
- **拡張**: 期間が「暦月以外」（週次・ローリング等）になる制限を足す場合は、`period` の意味を拡張するか、別 `quota_key` を分ける（**第2段階**でよい）。

### 1.6 編集と既存の日次分析

- **方針**: 編集は本文・気分のみを更新する。**`analysis_results`・`emotion_tags` には一切触れない**（編集と分析は完全に分離）。
- **UX**: 分析のやり直しは **別操作**（種類 A / B）で行う。編集保存時に分析が消える旨の警告は不要。
- **理由のメモ**: 分析の再取得はユーザーが任意のタイミングで行うものであり、編集とは結合しない。

## 3. 影響範囲（ファイル・レイヤ）

### 3.1 API / サーバー

| 対象 | 内容 |
|------|------|
| `app/api/v1/entries/[id]/route.ts` | `PATCH`（本文・任意 mood）。本人・暗号化・バリデーション。Gemini なし |
| `app/api/v1/analysis/generate/route.ts` | 日次: A/B 分岐、B は Deep・**同一週判定**・クォータ。`posted_at` は timestamptz 対応を継続 |
| `lib/date-utils.ts` 等 | **`getWeekRangeInTokyo()`** と `posted_at` の日付比較（同一週に含まれるか） |

### 3.2 DB

- **`public.quota_usage`**（`docs/supabase-migration.sql`）。種類 B は `quota_key` で識別し、**`analysis_results` にフラグを足して集計する方式は採用しない**（初回保存と B の区別が曖昧になるため）。
- アプリは **サービスロール**（既存と同様）で読み書き。RLS は **全拒否**のサーバ専用ポリシー。

### 3.3 フロント

| 対象 | 内容 |
|------|------|
| `journal-page.tsx` | 未取得時の **A** と取得済みの **B** で UI 分離。B は Deep のみ＋文言 |
| `history-page.tsx` 等 | 全期間 **編集**。**B** は**同一週内**・Deep のみ |

### 3.4 テスト

- Vitest: `PATCH`、日次生成の A/B・429・**週境界**（月曜／日曜）
