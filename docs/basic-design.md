# Reflect 基本設計書（詳細版）

concept.md / design.md を統合し、実装に落とし込むための基本設計を定義する。

---

## 1. プロジェクト定位

### 1.1 製品定義

| 項目 | 内容 |
|------|------|
| 製品名 | Reflect |
| 種別 | 自己分析アプリ（日記 × AI 分析） |
| 本質 | 「自分というブラックボックス」をログと文章で可視化する |
| 差別化 | 記録ではなく「解釈・気づき・言語化」をAIが担う |

### 1.2 設計原則（外さないこと）

1. **AIの質**  
   - 占い・スピリチュアル・ふわっと励まし禁止  
   - 「傾向」「パターン」「先週も出てた」のような**冷静で刺さる分析**に限定  

2. **コスト制御**  
   - 「AIを使うアプリ」ではなく「AIコストを制御するアプリ」  
   - Free: 超軽い（分類・短文）。Deep: 重い（分析・要約・生成）  

3. **価値の指標**  
   - 分析結果をスクショして誰かに送りたくなるか → Yes を目指す  

---

## 2. ユーザーとペルソナ

### 2.1 メインターゲット（設計の主軸）

- **年齢**: 20〜45歳（concept 30〜45 / design 20〜40 を統合）
- **属性**: 知的ホワイトカラー、自己啓発・内省好き
- **ペイン**: 自分を変えたいが方法がわからない、モヤモヤはあるが言語化できない
- **行動**: ChatGPTに人生相談したことがある層

### 2.2 サブターゲット

- メンタル安定を求めているがカウンセリングまでは行かない層
- 転職・キャリアで悩んでいる層

### 2.3 ユーザー区分と目的

| 区分 | 目的 | 期待する体験 |
|------|------|--------------|
| Free | 習慣化・価値実感・トークンコスト固定 | 書くだけで少しずつ傾向が見える |
| Deep | 「自分の取扱説明書」・ガチ分析 | 人格サマリー・深層分析・問いかけ |

## 3. 機能要件（詳細）

### 3.1 共通基盤

- **認証**: **メール + Google** のみ（OAuth は Google）。実装は Supabase Auth 等で一意ユーザー識別。
- **データの所在**: ユーザーごとに日記・分析結果は完全分離（マルチテナント）

### 3.2 Free 機能仕様

#### 3.2.1 日記入力

| 項目 | 仕様 | 備考 |
|------|------|------|
| 頻度 | 1日1回まで | リセットは 0:00 JST 等、タイムゾーン要定義 |
| 文字数 | 最大 500〜800 文字 | 超過時は UI で制限 or 切り捨て |
| 入力内容 | 自由記述 | プロンプトで「今日の出来事・気持ち」を誘導してもよい |
| 気分 | 任意 or 必須の単一選択 | 「気分を選ぶ」を concept に合わせて実装 |

#### 3.2.2 AI 簡易分析（Free）

- **実行タイミング**: 日記保存時 or バッチでまとめて（コスト・レイテンシのトレードオフ要検討）
- **出力項目（固定）**  
  - 今日の感情タグ: 3語程度（例: 不安・焦り・期待）  
  - 思考タイプ: 1つ（例: 不安傾向・他責傾向・完璧主義傾向）  
  - 一言コメント: 1文（占い・ポエム禁止のプロンプトで制御）
- **深掘り禁止**: 段落・長文解説は Deep のみ（コスト制御）

#### 3.2.3 履歴・感情ログ

- **7日分履歴**: カレンダー＋テキスト一覧。編集・削除の要否を仕様で決定
- **感情ログ**: 日ごとの感情タグを時系列で表示（「不安/怒り/喜び/焦り」等のタグ辞書を定義）

### 3.3 Deep 機能仕様

#### 3.3.1 入力の拡張

- 文字数制限解除（長文可）
- 1日複数投稿可（回数上限は別途設定可能にしておく）

#### 3.3.2 深層分析

- **思考パターン分析**  
  - 認知の癖、ネガティブループ検出、行動トリガー特定  
  - 週 or 月単位のレポートを生成（concept の「週/月にAI分析」に合わせる）  
  - **実行タイミング**: 原則、**ユーザーの操作起点**（例: 「週次サマリを見る」「月次レポートを見る」ボタン）で生成する（重いバッチを避ける）  
  - **年単位**: 設計・コストが許容できる場合に年次でも実行（後述の年次レポートと連動）
- **感情推移グラフ**  
  - 週・月単位で可視化  
  - メンタルの波が分かる粒度（日/週/月の切り替え）  
  - **年単位**: 同上。設計（集約粒度・UI）とコスト（トークン/集計負荷）がクリアできる場合に年間ビューを追加

#### 3.3.3 人格サマリー（キラー機能）

AIが生成するドキュメントとして固定:

- あなたの傾向まとめ
- 強み / 弱み
- 落ち込みやすい条件
- 回復しやすい行動

- **更新頻度**: 月1回等、データ量とコストで決定
- **保存形式**: テキスト＋必要ならPDFエクスポート

#### 3.3.4 年次レポート（条件付き）

- **内容**: 1年分の傾向まとめ・感情推移・思考パターンの年間サマリー。人格サマリーの年間版＋PDF出力（concept の「年間PDF人生レポート」に対応）。
- **生成方法（トークン節約）**: 生の日記テキストは渡さない。**それまでに出力した週次・月次サマリの圧縮版**（要約やキーワード化したもの）を入力として AI に渡し、年間サマリを生成する。必要に応じて感情ログの集計値など軽量なメタデータのみ追加する。
- **前提条件**: 次の2点を満たす場合に提供する。
  1. **設計**: 年単位の集約粒度・要約方針・UI（年間ビュー・PDFレイアウト）が一貫して定義できること。
  2. **コスト**: 年1回の重い分析を Deep 単価に収められること（トークン上限・実行タイミング・キャップの見積もり）。
- 条件がクリアできない間は「月次＋手動で振り返り」にとどめ、年次は将来対応とする。

#### 3.3.5 問いかけモード

- 例: 「最近◯◯が多いけど、なぜだと思う？」
- AIがカウンセラー的に質問を生成
- 実行頻度・1回あたりの質問数はコストとUXで設計

### 3.4 concept との対応整理

| concept | design / 本設計での扱い |
|---------|-------------------------|
| 週1サマリー | Free の「7日分＋簡易分析」で対応 or **週次サマリ閲覧時に生成（操作起点）** |
| 感情トレンド可視化 | Free の感情ログ＋Deep の感情推移グラフ |
| 月次AI分析レポート | Deep 深層分析＋人格サマリーの一部 |
| 思考タイプ分類 | Free で1種、Deep で複数・深掘り |
| ストレス源ランキング | Deep の思考パターン／行動トリガーから導出 |
| 年間PDF人生レポート | Deep の週/月サマリの圧縮版を入力に年間サマリを生成＋PDF（生データは渡さない） |

## 4. 非機能要件

### 4.1 パフォーマンス

- 日記保存後、Free の簡易分析は **数十秒以内** に表示（目標値は別途）
- 一覧・カレンダーは **2秒以内** 表示を目安

### 4.2 コスト設計（AI）

| 区分 | 目安 | 施策 |
|------|------|------|
| Free | 1日あたり 1,000 トークン以下 | 分類・短文のみ、深掘り禁止、モデルは軽量 |
| Deep | 1人あたり 月数十〜百円程度 | **週/月（操作起点）生成＋キャッシュ**、差分分析、文字数制限・回数制限でキャップ |

- 無料ユーザーが増えても **月数百円レベルで数百人** を許容できる設計にする

### 4.3 セキュリティ・プライバシー

- 日記・分析結果は **ユーザー本人のみ** アクセス可能
- データは **暗号化保存**（at rest）と **HTTPS**（in transit）を前提
- 個人を特定しない形での **分析・改善用ログ** の可否をポリシーで定義

### 4.4 可用性・運用

- MVP では可用性目標は「可能な範囲で」とし、SLA は後から定義可
- ログ・モニタリングで **AI失敗・レイテンシ悪化** を検知できるようにする

## 5. システム構成（概要）

### 5.1 コンポーネント

- **フロント**: Web アプリ（SPA or SSR は技術選定で決定）
- **バックエンド**: API サーバ（認証・日記CRUD・分析トリガー・配信）
- **AI**: 外部 LLM API（プロンプト・トークン制限・モデル切り替えを実装側で管理）
- **ストア**: ユーザー、日記、分析結果、サブスク状態を永続化

### 5.2 データの流れ（コア）

1. ユーザーが日記＋気分を送信
2. API が保存し、Free の場合は「簡易分析」ジョブをキック（同期的でも非同期でも可）
3. AI 結果を保存し、フロントに返す or プッシュ
4. Deep は週/月/年レポートを**操作起点**で生成し保存（必要なら非同期ジョブ）
5. フロントは日記一覧・カレンダー・感情ログ・分析レポートを API から取得して表示

### 5.3 技術思想の反映

- **日記ごとにフル分析しない**: Free は短文のみ、Deep は週/月単位
- **差分分析**: 前回からの差分を渡してトークン節約（設計で検討）
- **年次は既存サマリの圧縮版を渡して生成**: 年間レポートは生の日記を渡さず、週/月サマリの要約・圧縮版を入力とする（トークン節約）
- **モデル切り替え**: 軽いタスクは軽量モデル、重い分析は高能力モデル（コストと品質のバランス）

### 5.4 「気分」と「感情ログ（感情タグ）」の違い（設計判断）

- **気分（Mood）**: ユーザーが自己申告で選ぶ単一値（例: happy/neutral/down/anxious）。入力が速く、習慣化に寄与する。
- **感情ログ（EmotionTag）**: 日記本文からAIが推定する複数タグ（例: 不安・焦り・期待）。推移の可視化や傾向分析に向く。

**区別する必要がある（推奨）理由**

- **自己申告 vs AI推定のズレ**が洞察になる（例: 気分は neutral だが本文は不安タグが多い、など）。
- Free は気分だけでも成立する一方、Deep の分析精度・説得力は感情タグがあると上がる。

**区別しない（統合）案**

- 実装を軽くしたい場合は、気分を感情タグの一種として扱い「source=user/ai」を持つ単一テーブルに統合できる（ただし本設計書では当面は分離で記述）。

## 6. データモデル（概念）

### 6.1 エンティティ

 - **User**: id, 認証子（メール or OAuth id）, プラン（free / deep）, 作成日時 等
- **Entry（日記）**: id, user_id, 本文, 文字数, 投稿日（日単位）, 作成日時 等
- **Mood（気分）**: id, user_id, entry_id or 日付, 選択値, 日時 等
- **AnalysisResult（分析結果）**: id, user_id, 対象範囲（日付 or 期間）, 種別（daily / weekly / monthly / personality）, 生データ（JSON 等）, 作成日時
- **EmotionTag（感情タグ）**: 日付・entry に紐づくタグ（不安/怒り/喜び/焦り 等）、辞書を定義
- **Subscription**: user_id, プラン, 課金状態, 期限 等（決済基盤に依存）

### 6.2 分析結果の種別

| 種別 | 対象 | 主な内容 |
|------|------|----------|
| daily | 1日1エントリ | 感情3語、思考タイプ1つ、一言コメント（Free） |
| weekly | 直近7日 | 思考パターン、感情推移、短文サマリー（Deep） |
| monthly | 直近30日 | 深層分析、感情推移グラフ用データ、ストレス源等（Deep） |
| yearly | 直近1年 | 年次レポート用。入力は週/月サマリの圧縮版（生日記は使わない）。設計・コストが許容できる場合のみ（Deep） |
| personality | 蓄積データ | 人格サマリー（傾向・強み/弱み・落ち込み条件・回復行動） |
| question | 任意 | 問いかけモード用の質問文 |

## 7. 画面・フロー（概念）

### 7.1 主要画面

- **ホーム / 今日の日記**: 入力フォーム、気分選択、保存、直近の簡易分析表示
- **履歴 / カレンダー**: 7日分（Free） or 全期間（Deep）、日付クリックで詳細
- **感情ログ**: 時系列の感情タグ一覧（Free は簡易、Deep はグラフ付き）
- **分析レポート**: 週/月レポート、人格サマリー、問いかけ、年次レポート（条件を満たす場合）（Deep）
- **設定**: アカウント、通知、プラン・課金（Deep 解約等）

### 7.2 コアフロー（concept 準拠）

1. 今日の出来事を書く（300〜800字を目安に誘導）
2. 気分を選ぶ
3. 保存して「何も考えず閉じる」でよい
4. 必要になったら「週/月（必要なら年）でAI分析レポートを見る」（操作起点で生成し、生成済みはキャッシュ）

---

## 8. 課金・収益

- **Free**: 0円、習慣化と価値実感
- **Deep**: 月額 980円程度（1,000円を超えない）
- 成立条件: サブスクのみ想定。広告・買い切りは想定しない（concept の通り）

---

## 9. 今後の検討事項（実装前に詰めるもの）

- 認証（メール＋Google）の具体（Supabase Auth で十分か、独自実装か）
- 日記保存と簡易分析の「同期 vs 非同期」と UX（ローディング・エラー表示）
- 感情タグ・思考タイプの**辞書とプロンプト**の具体化（占いっぽくならない文言）
- 週/月/年レポートの **操作起点生成** の UX（生成ボタン、生成中表示、キャッシュ、再生成ポリシー）
- 人格サマリーの初回生成条件（何日分溜まったら生成するか）
- 年次レポートの可否判断：設計（年間集約・PDF仕様）とコスト見積もりが揃ったタイミングで提供開始
- 問いかけモードのトリガー（日付ベース / パターン検出ベース）
- ストレージ・DB 選定（RDB / Document / 混在）
- 監視・アラート（AI API 障害、レイテンシ、コスト超過）
