# 管理画面（Admin）設計案

## 1. 目的

- **運用・サポート**で必要な操作（ユーザー状況の確認、課金トラブルの一次切り分け、問い合わせ対応の文脈確認）を、**一般ユーザー向け UI とは分離**して安全に行う。
- 現状は **一般ユーザーと同じ `users` テーブル・同じ NextAuth** のみで、**管理者という概念がコードにも DB にも無い**。本設計はその前提を変えるための方針を示す。

## 2. 設計原則

| 原則 | 内容 |
|------|------|
| **分離** | ルートは `/admin/*` にまとめ、ナビ・レイアウトをダッシュボード用と完全に分ける。 |
| **サーバー中心** | 一覧・検索・更新はすべて **Route Handler / Server Actions + `createSupabaseAdminClient()`**。クライアントにサービスロール鍵を渡さない。 |
| **最小権限** | 「誰が管理者か」を明示し、**環境変数だけ**または **DB フラグ**のどちらか一方に統一する（後述）。 |
| **監査** | 重要操作は **ログ（構造化）＋必要なら `admin_audit_log` テーブル**に残す。 |
| **本番のみ** | ローカルでは `ADMIN_*` 未設定なら管理画面ルートは 404 またはメッセージのみにし、誤公開を防ぐ。 |

## 3. 管理者の定義（採用候補）

どちらか **1 方式に決めて**実装する（混在させない）。

### 案 A: 環境変数によるメール許可リスト（MVP に向く）

- `ADMIN_EMAILS`（カンマ区切り）に含まれるメールのユーザーだけが管理画面に入れる。
- **利点**: マイグレーション不要。実装が早い。  
- **欠点**: 追加・削除はデプロイ or 環境更新が必要。複数人のローテーションは運用でカバー。

### 案 B: DB の管理者フラグ（拡張に向く）

- `users.is_admin BOOLEAN DEFAULT false`、または `admin_operators(user_id)` のような小さなテーブル。
- **利点**: デプロイなしで権限付与・剥奪可能。  
- **欠点**: 「最初の管理者を誰が立てるか」は SQL かシードが必要。

**推奨**: まず **案 A で MVP** → 利用頻度と人数が増えたら **案 B に移行**（移行時は案 A をフォールバックとして残すか、一度だけ SQL で `is_admin` を埋める）。

## 4. 認可の置き場所

1. **Middleware**（`middleware.ts`）  
   - `/admin` プレフィックスにマッチしたリクエストだけ、セッション必須・管理者チェックへ誘導。  
   - 未ログイン → `/login?callbackUrl=/admin` など。  
   - ログイン済みだが管理者でない → **403** 専用ページ（「権限がありません」）。一般ダッシュボードへ自動リダイレクトしない（URL がバレにくい）。

2. **各 API / Server Action**  
   - Middleware をすり抜ける経路がないよう、**必ず `requireAdmin()` のような共通関数**で二重チェック（`auth()` + メール or DB）。

3. **NextAuth セッションへの `role` 載せ**（任意・第 2 段階）  
   - JWT に `isAdmin: boolean` を載せ、毎回 DB を叩かないようにする。トークン更新タイミングで再計算。

## 5. URL・情報設計（案）

| パス | 役割（案） |
|------|------------|
| `/admin` | ダッシュボード（KPI サマリの目安: 直近登録ユーザー数、Deep 契約ざっくり件数など ※後述） |
| `/admin/users` | ユーザー一覧・検索（email / id / 論理削除フラグ） |
| `/admin/users/[id]` | ユーザー詳細: `users` + `subscriptions` のメタ、**本文は原則マスク or 閲覧オプトイン** |
| `/admin/contact` | `contact_requests` の一覧（問い合わせ対応） |
| `/admin/system` | 環境・バージョン・Stripe モード表示のみ（編集なし）など |

**第 1 段階では「閲覧中心」**とし、**削除・課金操作・本文の全文表示**は別イシューで要件を固める（後述「スコープ外」）。

## 6. 画面で触るデータ（既存スキーマとの対応）

- `users` … plan / deleted_at / email / name  
- `subscriptions` … Stripe 連携の整合確認用  
- `contact_requests` … サポート用  
- `entries` / `analysis_results` … **個人情報・機密度が高い**ため、管理画面では  
  - **件数・日付・暗号化フラグ程度**に留める、または  
  - **「本文を表示する」は別権限・別ログ必須**とする  

暗号化（`ENCRYPTION_KEY`）があるため、**管理画面で本文を復号して表示する**場合は、**利用規約・社内ポリシー・ログ**をセットで決める。

## 7. API 設計（案）

- `GET /api/admin/users?q=&page=` … 検索・ページング  
- `GET /api/admin/users/[id]` … 詳細メタ  
- `GET /api/admin/contact-requests` … 問い合わせ一覧  

すべて **`requireAdmin()`** 内で 401/403 を返し、**一般の `/api/v1/*` とは名前空間を分ける**。

## 8. 監査ログ（推奨）

テーブル案（必要になったら追加）:

- `admin_audit_log(id, actor_user_id, action, target_type, target_id, meta JSONB, created_at)`  
- 例: `action = 'user.view' | 'contact.update'` など。

MVP では **構造化ログ（`logger.info`）のみ**でも可。ただし本番の問い合わせ対応では DB ログがあると強い。

## 9. セキュリティチェックリスト

- [ ] 管理 API に **Rate limiting**（Edge / アプリ層）を検討  
- [ ] **CSRF**: Server Actions 利用時は Next の既定対策 + 同一オリジン確認  
- [ ] **検索クエリ**の SQL インジェクション対策（Supabase のパラメータバインド徹底）  
- [ ] **本番 URL** をロボット・検索エンジンから除外（`robots.txt` / `noindex`）  
- [ ] **2FA** は将来検討（管理者アカウントが Google ログインのみならリスク低減）

## 10. 実装フェーズ案

| フェーズ | 内容 |
|----------|------|
| **0** | 設計レビュー・管理者の定義（案 A or B）確定 |
| **1** | `/admin` レイアウト + Middleware + `requireAdmin` + 空ダッシュボード |
| **2** | ユーザー一覧・検索（メタのみ、本文なし） |
| **3** | `contact_requests` 一覧・詳細 |
| **4** | 監査ログ、必要なら本文閲覧（別権限） |

## 11. あえて第 1 段階ではやらないこと（スコープ外）

- 一般ユーザー向け画面からの **管理者昇格 UI**  
- **Stripe Dashboard の代替**（返金・請求書の詳細操作は Stripe に任せる）  
- **ジャーナル本文の一覧検索**（コンプライアンス確定まで保留が無難）

## 12. 未決事項（決めると実装が楽）

1. 管理者は **案 A（環境変数）** で開始するか、最初から **案 B（DB）** か。  
2. 管理画面のログインは **既存の Google / メールアカウントのまま**でよいか（推奨: そのまま + 許可リスト）。  
3. KPI（`/admin` トップ）に **何を載せるか**（例: 直近 7 日の新規ユーザー数、問い合わせ未読数など）。
